# Syncing this fork with upstream

How to bring new commits from `grandamenium/cortextos` into `saurav-yirifi/sb-cortextos-fork` without losing local patches.

## Repo topology

```
upstream  grandamenium/cortextos              (read-only fetch)
   │
   │ fork
   ▼
origin    saurav-yirifi/sb-cortextos-fork     (push + pull)
   │
   │ clone
   ▼
local     /Volumes/MacStorage/.../sb-cortextos-fork
```

`upstream` push is disabled to prevent accidents (`git remote -v` shows `upstream → DISABLED` for push).

Local patches that diverge from upstream:
- `be445f0` — `fix(auth): make cookie Secure flag env-controlled` (PR #1, merged into our fork's `main`).

## Quick status check

```bash
cd /Volumes/MacStorage/UserData/0devprojects/sb-cortextos-fork
git fetch upstream
git rev-list --left-right --count main...upstream/main
# Output: <our-ahead> <upstream-ahead>
# 1 0  → we have the auth patch, upstream is idle
# 1 5  → upstream has 5 new commits to merge in
```

If the right number is 0, nothing to do.

## Path A — Manual git (most control)

```bash
git checkout main
git fetch upstream
git log --oneline main..upstream/main      # preview what's coming
git merge upstream/main                    # creates merge commit
# Resolve conflicts if any (see "Conflicts" below)
# Push via PR — direct main push is blocked by hook
git checkout -b sync/upstream-$(date +%Y-%m-%d)
git push origin HEAD
gh pr create --repo saurav-yirifi/sb-cortextos-fork --base main \
  --title "sync: merge upstream/main" \
  --body "Brings in upstream commits since last sync."
gh pr merge <num> --merge --delete-branch --repo saurav-yirifi/sb-cortextos-fork
git checkout main && git pull origin main
```

Use `--merge` (not `--squash`) on sync PRs so individual upstream commits stay legible in our history.

## Path B — GitHub "Sync fork" button

On https://github.com/saurav-yirifi/sb-cortextos-fork there is a **"Sync fork"** button. One click, GitHub does the merge server-side. Then locally:

```bash
git checkout main && git pull origin main
```

Caveats:
- Bails out if there are conflicts → fall back to Path A.
- Never click "Discard commits and match upstream" — that wipes our local patches (e.g. `be445f0`).

## Path C — Banner-driven sync (Telegram-approved)

Banner (the analyst agent) has a built-in upstream-sync skill. If `ecosystem.upstream_sync.enabled: true` in `orgs/sb-personal/agents/analyst/config.json`, Banner runs `cortextos bus check-upstream` on a schedule and pings on Telegram with a plain-English summary of new upstream commits.

```bash
# Inside an agent session, or run manually:
cortextos bus check-upstream             # detects + summarizes, never merges
cortextos bus check-upstream --apply     # only after explicit "yes" reply
```

Workflow:
1. Banner detects new upstream commits, posts to Telegram.
2. We read the summary, can ask follow-up questions.
3. Reply "yes" / "no" / "show me X file diff".
4. On "yes" Banner runs `--apply`, then verifies system health.

Safety rules baked in:
- Never auto-merges — explicit approval required every time.
- Never merges during night mode.
- Markdown template changes are ADD-ONLY — won't overwrite our customizations.
- Conflicts surface back to us with explanation.

This is the preferred day-to-day path.

## Conflicts

Most likely conflict: `dashboard/src/lib/auth.ts` (our `AUTH_COOKIE_SECURE` patch and upstream's cookie config can collide).

Steps:

1. After `git merge` stops on conflict:
   ```bash
   git status                   # lists conflicted files
   ```
2. Open the file. Find `<<<<<<<` markers. Pick the version that **keeps `AUTH_COOKIE_SECURE`** but integrates any new upstream changes (new cookie types, new options).
3. If upstream has merged a similar fix and ours is now redundant, take the upstream version and delete ours.
4. Stage and continue:
   ```bash
   git add dashboard/src/lib/auth.ts
   git commit                   # finalises the merge
   ```
5. Continue with the PR step from Path A.

## When upstream re-issues `cortextos ecosystem`-style files

`ecosystem.config.js` in this repo is auto-generated by `cortextos ecosystem`, with hardcoded absolute paths to whichever machine ran it last. **Never commit a regenerated `ecosystem.config.js`** — it leaks our local SSD paths and breaks on every other machine. Re-run `cortextos ecosystem` locally after any sync, but don't stage it.

If upstream's portable version of `ecosystem.config.js` evolves, take their changes; regenerate ours locally afterwards.

## Direct push to `main` is blocked

The CLAUDE.md rule "All changes via PR, no direct writes to main" is enforced by a hook. Even on this personal fork, every change goes via:

1. Branch (`fix/*`, `chore/*`, `sync/*`)
2. Push branch
3. `gh pr create --repo saurav-yirifi/sb-cortextos-fork`
4. `gh pr merge <num> --squash --delete-branch --repo saurav-yirifi/sb-cortextos-fork` (use `--merge` for sync PRs)
5. `git checkout main && git pull origin main`

## Useful one-liners

```bash
# How far behind/ahead are we?
git fetch upstream && git rev-list --left-right --count main...upstream/main

# What did upstream change that we haven't merged?
git fetch upstream && git log --oneline main..upstream/main

# What did we change that upstream doesn't have?
git fetch upstream && git log --oneline upstream/main..main

# Show the file-level diff between our main and upstream's
git fetch upstream && git diff upstream/main..main --stat
```
